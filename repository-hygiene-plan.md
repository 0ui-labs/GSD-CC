# Repository Hygiene Implementation Plan

## Goal

Clean up repository hygiene issues without hiding legitimate project files.

The target outcome is:

- committed noise files are removed from version control
- clearly generated local artifacts are ignored
- ignore rules stay narrow and intentional
- future contributors can tell which files are source and which are disposable

## Why This Change Comes Sixth

The earlier phases focus on safety, correctness, and documentation truth.
After that, repository hygiene is the next practical improvement because it
reduces day-to-day friction in the worktree and lowers the chance of accidental
commits.

This is lower risk than installer safety, task-plan consistency, or auto-mode
Git correctness, but it still matters because noisy repos make every other
workflow harder to trust and maintain.

## Current Problems

1. Multiple `.DS_Store` files exist inside the repository tree and some appear
   to be tracked project noise rather than intentional assets.
2. A local `.claude/` directory can be created by project-local install flows
   but is not currently ignored at the repo root.
3. The repo already ignores a few generated files, but the policy is implicit
   and incomplete.
4. There is no documented rule for deciding whether a new artifact should be
   versioned, ignored, or removed.
5. Hygiene fixes must avoid masking real source files or user-created project
   inputs.

## Non-Goals

- Broadly ignore entire source directories.
- Hide ambiguous files whose ownership or purpose is not yet understood.
- Replace careful artifact classification with blanket ignore patterns.

## Hygiene Principles

1. Remove noise from version control when it is clearly machine-generated.
2. Ignore only files that are reproducibly generated and safe to discard.
3. Do not ignore files just because they are inconvenient in `git status`.
4. Prefer precise repo-root patterns over broad recursive globs when possible.
5. If a file might be meaningful project input, classify it before ignoring it.

## Artifacts To Address

### 1. Finder metadata

Observed examples include:

- `./.DS_Store`
- `./gsd-cc/.DS_Store`
- `./gsd-cc/skills/.DS_Store`
- `./gsd-cc/skills/seed/.DS_Store`
- `./gsd-cc/skills/seed/types/.DS_Store`

These should not live in the repository and should be removed from version
control where tracked.

### 2. Local Claude installation output

Observed example:

- `./.claude/`

This is a local runtime/install artifact for project-local setups, not source
code. It should be ignored at the repo root unless the project deliberately
commits a template or fixture under a different path.

### 3. Generated runtime state

Current ignore rules already cover some generated files such as:

- `*.log`
- `.gsd/auto.lock`

This phase should review whether additional generated runtime files belong in
`.gitignore`, but only when they are both:

- clearly generated by the toolchain
- never intended as durable project artifacts

Candidates for classification during implementation:

- `.gsd/auto.log`
- `.gsd/COSTS.jsonl`
- similar local execution residue, if it is not meant to be committed

These should only be ignored if the product behavior says they are operational
artifacts rather than part of the intended project record.

## Proposed Implementation

### Phase A: Classify artifacts

Create a simple classification pass for repo noise:

- tracked machine-generated files to delete from Git
- untracked local-only files to ignore
- ambiguous files to leave untouched until their role is confirmed

This avoids accidentally hiding real content.

### Phase B: Remove committed noise

Remove committed Finder metadata files from the repository and ensure the
existing `.DS_Store` ignore rule remains in place.

This phase should focus on exact known files, not broad destructive cleanup.

### Phase C: Add narrow ignore rules for local install output

Add targeted ignore coverage for project-local install output, most likely:

```text
/.claude/
```

Use a root-anchored rule so we ignore only the local install directory created
in this repository, not any unrelated nested path that might one day be a real
fixture or example.

### Phase D: Review generated runtime files

For each candidate runtime artifact, answer:

1. Is it generated automatically?
2. Is it safe to delete and regenerate?
3. Is it intended for users to inspect or commit?
4. Would ignoring it hide useful project state?

Only after those questions are answered should additional ignore rules be added.

### Phase E: Document the hygiene policy

Add a short contributor-facing note explaining:

- local install output belongs in `.gitignore`
- machine-generated Finder metadata should never be committed
- ambiguous files must be classified before adding ignore rules

This reduces future drift and keeps hygiene fixes consistent.

## Ignore Rule Strategy

### Rules we likely want

- keep `.DS_Store`
- add `/.claude/`

### Rules we should avoid unless justified

- recursive `.claude/` ignores without root anchoring
- broad ignores for `.gsd/*.md`
- broad ignores for skill or template directories
- catch-all ignores for Python, JSON, or Markdown files used by source content

## Files Expected To Change

- `.gitignore`
- `CONTRIBUTING.md` if we add a hygiene policy note

Likely file removals from version control:

- tracked `.DS_Store` files inside the repo tree

Possible follow-up changes:

- docs that mention local install behavior if `.claude/` should be called out
  as a local artifact

## Verification Plan

### Static verification

- confirm `.DS_Store` is ignored
- confirm the repo-root `.claude/` directory is ignored
- confirm ignore rules remain narrow and do not match source paths

### Manual smoke tests

1. Create or keep a local `./.claude/` directory and confirm `git status`
   stays clean for that path.
2. Confirm committed `.DS_Store` files are removed from Git tracking.
3. Create a nested path that should remain versionable and confirm it is not
   accidentally ignored by a broad rule.
4. Review any candidate `.gsd` runtime files and confirm each new ignore rule,
   if added, is justified by product behavior.

### Success criteria

- Finder metadata no longer appears as versioned project content
- local install output does not pollute `git status`
- ignore patterns remain explicit and conservative
- contributors have a clear rule for future hygiene fixes

## Commit Strategy

Planned atomic commits:

1. remove tracked `.DS_Store` artifacts
2. add narrow ignore coverage for repo-local generated outputs
3. document the hygiene policy for contributors

## Open Questions

1. Whether `.gsd/auto.log` and `.gsd/COSTS.jsonl` are operational residue or
   intended project artifacts.
2. Whether any committed `.DS_Store` files are present only locally or already
   part of the canonical branch history.
3. Whether the local install flow should also document that `./.claude/` is a
   disposable generated directory.

## Exit Criteria

This phase is done when:

- machine-generated noise files are no longer tracked
- repo-local install artifacts are ignored cleanly
- ignore rules are still precise enough to avoid hiding real source files
- contributors have a lightweight, explicit hygiene policy to follow
